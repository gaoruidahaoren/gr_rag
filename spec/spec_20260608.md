- `/api/chat` → `smartRewrite` → `rewriteQuery` (LLM) / `fallbackExtract` → `extractEntityKeywords`

**散落在**：`indexManager.ts` (运行时检测), `index_state.json` (构建时快照), `chunks_meta/` (分片存储)

## 5. 可疑行为清单

### 5.1 `index_state.json` 不存在时增量构建的行为
`buildIncremental.cjs` 首次运行且 `index_state.json` 不存在时，会将**所有文件标记为"新增"**，然后执行增量追加逻辑。这意味着首次增量构建 = 全量构建，但走的是增量路径（追加而非覆盖），可能导致 LanceDB 表中有重复数据。**全量构建有 `DROP TABLE` 逻辑，增量没有。**

### 5.2 全量构建写 `index_state.json`，但增量构建覆盖它
全量构建末尾会保存 `index_state.json`，增量构建也会在最后更新它。两者格式一致，但增量构建不验证现有的 index_state.json 是否来自同一次构建。如果全量构建后手动删除了某个分片 JSON，增量构建无法感知。

### 5.3 `vectorSearch` 返回的 chunk 在 RRF 中被实体过滤时，排名重新编号
`rrfFusion()` 中：被实体关键词过滤的 chunk 的 `vectorRank` 设为 null，但 `effectiveVecRank` 会跳过它继续计数。这意味着**后续 chunk 的实际 RRF 排名会比原始排名更靠前**。这是一种"缩进排名"策略，可能不符合直觉预期。

### 5.4 `hybridSearchExpanded` 公开但无调用方
这个函数在 `hybridSearch.ts` 中导出，但没有任何地方调用它。实际 Small-to-Big 扩展是通过 `ragEngine` 内部调用 `expandToParentContext` 实现的，绕过了这个包装函数。

### 5.5 `ragChat` 公开但无调用方
`ragEngine.ts` 导出了 `ragChat` (非流式版本)，但只有 `ragChatStream` 被 `/api/chat` 使用。两者共享大部分逻辑但独立维护。

### 5.6 `isVectorReady` 公开但无调用方
`vectorEngine.ts` 导出了 `isVectorReady`，但 `indexManager.isIndexReady` 自己检测 LanceDB 目录是否存在，不依赖这个函数。

### 5.7 会话管理的 "上一轮 query" 在追问时不更新
`chat/route.ts` 第 50-57 行：追问时用 `lastSearchResults.query` 作为上下文，但这个 query 始终是**上上一轮**的原始 query。如果用户连续追问 3 轮（第1轮问A，第2轮追问B，第3轮追问C），第3轮拿到的 `previousQuery` 仍然是 A 而非 B。因为 `saveLastSearchResults` 只在检索完成后才更新。

### 5.8 对话压缩是异步且不阻塞请求
`chat/route.ts` 第 60 行：`compressConversation(session.id, ...).catch(() => {})` — 异步触发，不 await。如果压缩还没完成就来了下一个请求，压缩结果不会被当前请求使用。且压缩失败被静默吞掉。

### 5.9 `entityRouter.routedSearch` 中实体召回后可能重复调用 hybridSearch
`routedSearch` 的 `smartRoute` 返回 `'hybrid'` 时会调用 `loadStructDocChunks` + `hybridSearch` 合并；但如果 `smartRoute` 失败，降级路径会再次调用 `hybridSearch`。存在同一 query 两次检索的风险。

### 5.10 `index_state.json` 用 MD5 hash，但文件编码可能不一致
`buildIndex.cjs` 和 `buildIncremental.cjs` 都读取文件内容做 MD5，但如果文件在不同操作系统/编码下保存，MD5 可能不一致，导致误判为"已修改"。

### 5.11 父文档缓存永不过期
`parentDocsCache` 在 `hybridSearch.ts` 中懒加载后永不过期。如果增量构建更新了 `parents.json`，运行中的服务不会感知，必须重启。

### 5.12 chunks_meta 分片数量硬编码
构建脚本中的分片逻辑和运行时 `bm25Engine.ts` 中的分片读取逻辑独立维护。如果构建脚本改了分片数但运行时没改，会导致读取失败或遗漏数据。

### 5.13 `isFollowUpQuery` 的追问检测只在 chat 中使用，search API 不支持多轮
`/api/search` 没有 session 概念，无法检测追问。

### 5.14 实体关键词过滤只在向量检索结果中生效
`rrfFusion` 中的 `vectorEntityFilter` 只过滤向量侧，不过滤 BM25 侧。逻辑上是合理的（BM25 本身就是关键词匹配），但如果 BM25 召回了不包含实体关键词的文档（比如 jieba 分词错误），不会被过滤。

## 附录：RRF 融合算法说明

当前 RRF（Reciprocal Rank Fusion）计算分数的完整逻辑：

```
RRF(d) = Σ 1/(k + rank_i(d))
```

其中 `k = 60`（平滑参数），`rank_i(d)` 是文档在第 i 个检索系统中的排名（从 1 开始）。

**核心流程**（`rrfFusion` 函数）：
1. **向量排名贡献**：遍历向量检索 top20 结果，排名从 1 开始，每个结果贡献 `1/(60 + effectiveVecRank)`
2. **BM25 排名贡献**：遍历 BM25 检索 top20 结果，排名从 1 开始，每个结果贡献 `1/(60 + rank)`
3. **融合排序**：同一 chunk 同时出现在两路结果中 → RRF 分数相加
4. **实体过滤**：向量检索中不包含实体关键词的 chunk，其 vectorRank 设为 null
5. **分数归一化**：`normalizedScore = 0.05 + (rrfScore - minRrf) / (maxRrf - minRrf) × 0.90`，映射到 0.05~0.95 区间
