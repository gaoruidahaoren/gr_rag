## 按 Issue 分类的补充检查

### 如果是 `[BUG]` 修复

额外确认：
- [ ] 是否补充了对应的回归测试？
- [ ] 修复是否引入新的边界问题？

### 如果是 `[DESIGN]` 设计变更

额外确认：
- [ ] 重构前后的行为是否等价？（通过现有测试验证）
- [ ] 是否有性能影响？（启动时间、检索延迟、内存占用）

### 如果是 `[API]` 公共接口变更

额外确认：
- [ ] 所有调用方是否已适配？
- [ ] 是否保留向后兼容？（旧调用方式是否仍可用）
- [ ] 迁移指南是否已写入 PR 描述？

### 如果是 `[MULTI]` 同根因修复

额外确认：
- [ ] 是否所有受影响模块都已统一修改？
- [ ] 统一后是否消除了重复逻辑？（diff 对比确认）
- [ ] 共享模块的接口是否清晰、可扩展？
